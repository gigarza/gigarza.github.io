JPMorgan: Web Data Verification

[ Created Summer 2019 ]

During my internship at JPMorgan, I was introduced with the problem a development team had of spending lots of hours doing manual labor comparing the data structure in their Oracle Database to a Data Models file they would receive from another team showing how the data should be moving through the pipeline. These manual checks were often extremely tedious, showing lots of errors that would only pop-up one at a time, requiring short and inefficient single fixes. Therefore, for the duration of my 10-week internship, I was tasked with creating a full-stack web application to solve this problem.

The web application itself was built using Java-Springboot and Angular. It allowed a user to upload a file of the correct type, and then select which models they wanted to compare. In our backend, we would go through the file, comparing each model against the database, collecting any errors that were found as we went. After the comparison logic was complete, the user would see a breakdown of the results on the front-end. This would show up categorized by model, with visual icons and coloring dependent on types of successes, warnings, or errors. Upon success, the website would also provide necessary SQL and Avro files that we auto-generated to help get a step-ahead on the next process. The web application was uploaded onto JPMorgan's internal cloud service, allowing immediate usage.

To The Beat

[ Created Summer 2018 ]

To The Beat is a web application that will make the ideal playlist for your runs, workouts, or even jams.

The front-end of this website was made with Bootstrap, CSS, and Spotify logos provided by Font-Awesome. The logic is generated by python/flask and javascript.

When the user first enters the application, they will click to log-in to Spotify which prompts Spotify API calls to authorize that user's account to be used. Once the account has been authorized with OAuth, the backend will receive the access token and the ID for the Spotify user.

Once redirected to the next page, the user will be prompted to enter their selections for the name and tempo they want for their new playlist. The backend will then consistently update what the user has typed. There is also a sound button on the page which plays a single beat at the speed they have entered. This is done by calculating how many milliseconds would be between each beat, then playing a simple single beat mp3 file.

Once the user clicks submit to enter their choices, the backend of the code then grabs the information from the user. It then makes Spotify API calls to first get all of the user's faved songs. Using the Spotify ID of each of these songs, it then makes another Spotify API call to get the track info, which has the tempo of the song. This tempo is then compared against the user's requested tempo, and if it is within 3BPM of the request, it is added to a list of songs for the new playlist.

Once the user's songs have been analyzed to find the ones which match tempo, a Spotify API call is made to create a new playlist with the name the user chose. The list of songs is then added into this new playlist. Lastly, the user is redirected to a result page where details about their new playlist are listed, and the playlist itself is embedded into the page.

SF Dispatches

[ Created Spring 2018 ]

This webapp acts as a way to represent various data about San Francisco Dispatches. The data was received through csv files, then input into a MongoDB database through a python backend file. Bootstrap/Bootswatch was used throughout the webapp to assist in frontend styling.

When the user first comes upon the site, they will hit the homepage, made solely through static html/css files. From this homepage, the user can then get to any other page on the webapp.

The first page they can go to is the Statistics & Graphs page. On this page, the user can find various graphs and facts about the data. The page itself was created using html/css files, but also an integration of Chartist.js to represent the data graphically. The data is connected into these files by looking for and grabbing the necessary information from the database in the main python file.

Informational Maps is the second page a user can choose to go to. The user has the option to checkout 4 different maps on this page for more visual information about the dispatch data. The first graph shows the zipcodes within San Francisco which seem to take the longest time to dispatch. The Heat Map shows the user the frequency of dispatch calls across the city. The third map shows the user which are considered the safest neighborhoods in the city (in relation to crime and dispatch call frequency). The final map then shows the zipcodes which have the highest increasing rate of dispatch calls. All of the information for these maps was figured out algorithmically through a python backend along with get requests on the database. The maps were then represented by using the Google Maps API and GeoJSON files.

Thirdly, a user can look at the Most Likely Dispatch page, which will show them the dispatch that is most likely to occur based on a location and time. The time input was integrated using TimePicker for jQuery, while the map and the marker drop were integrated using the Google Maps API. After the user has clicked submit, an algorithm begins to discover what range of time and what areas to grab data from. From there, the new list is gone through to find which type of dispatch call will be most likely, and outputs this result to the user.

Check out the code & readme on github to find out more information about the logic for the backend!

PetPals

[ Created Spring 2019 ]

Completed while taking the iOS Mobile Computing course at UT. This project's focus was to create a fully working iOS Mobile Application that would allow pet owners to meet other pet owners to set up play dates; basically, a tinder for pets.

To implement ability for users to create accounts, used Firebase. Firebase authentication then held all this information so users could quickly login to the app upon load, or be auto-recognized by their device. Additionally, Firebase was used as a database/storage tool in order to keep track of all user preferences/settings, as well as the users they've "swiped" or "matched with". Each user has their own profile page, where they can update their profile picture, name, animal type, and a short bio, and settings page where they can update their preferences on the types of matches they want to look for.

On the home screen for the application, users would see a single card at a time representing a different user which met the specified requests and settings. One of these settings included the distance between the users, which was grabbed by the most recent location of those users when having the app open, which was grabbed and then stored using Geofire. In order for the user to decide whether to "like" or "decline" this other user, they could simply swipe right or left on the screen. This was implemented using Koloda for the swipe animations and functionality.

Lastly, a user could go to a screen which would show all their recently created matches (users which both "liked" each other), and set-up pet play dates. After setting up dates, each user could then see their matches and pet dates categorized by upcoming scheduled dates, invitations for pet dates by other users, and tentative dates (dates this user suggested to others). A toggle at the top of this screen would then take the user to a calendar, implemented with JTAppleCalendar, which would show a visual of all the upcoming set dates.

Waitless

[ Created Summer 2018 ]


This webapp for mobile devices provides a Capital One user the ability to have a faster checkout experience at retail stores.

When the user opens the home page, and has already logged in, they have the option to add new items to purchase, look at their current cart, or view their transaction history. When they want to add a new item, they will go to the scan screen.

This scan screen recognizes the user's location, and initially gives the closest retail store. If this is not the location they are at, they can click to change through to other local retail stores until they find the correct one. This is all done using the Google Maps API to recognize a user's location and search for the top 10 closest retail stores. After the user has chosen a location, they place the barcode of their item under the camera. The program then scans the barcode, finds the product, and adds to the cart. The scanner itself was integrated using QuaggaJS, which will return the number of the barcode to the backend of the code. This barcode is then lookedup in the database to find the item that matches it. This item is then added into a list that represents the user's cart. Once the user is ready to complete their purchase, they can click checkout.

The checkout screen will then list all the items in the cart, also allowing the user to remove any unwanted items. Finally, they click submit purchase, and the total is automatically removed from their Capital One bank account using a call to the Capital One API.

CapGen

[ Created Summer 2018 ]

Created at the 2018 CDK Global Annual Hackathon, CapGen is an API in which a user can input a car's vin number, and receive an automatically generated description for the vehicle. For the hackathon, the project was presented on a UI, built in Angular, to simply demo the program's capabilities. It was imagined that if this were to go to production for the company, it would simply act as an API for clients that wouldn't have such a structured UI. The API itself was deployed using Docker, and was set to be as production ready as possible.

For the backend of the project, two separate models were trained in order to produce the descriptions. The model for the images was created using Keras' built in VGG model in order to extract features from the photo as a way to encode the images. For training, the correlated descriptions were tokenized and then the inputs and outputs were encoded in order to be prepared for the model. The model itself was then built in such a way that there were 3 portions: a photo feature extractor, a sequence processor, and a decoder. The model was then trained and fit using data provided by CDK global on AWS, and saved. This model is what predicts and generates the descriptions of the vehicles.

When presented, the UI has a user input a vehicle's unique vin number, which our API then calls a CDK Vin Decoder API in order to gather images and features of the specific vehicle to send to the backend. In the backend a function is called which sends this information to the pre-saved model and spits out the generated description. The UI then transitions to a results page where the image, features, and newly created description are displayed to the user.

At the time of the hackathon demonstration only the model for the images was being used, as the feature model portion was still being trained and developed. As this was a project done while working at CDK Global as an intern, no code is available to be seen. Film Prediction

[ Created Spring 2019 ]

Completed while taking a Introduction to Data Mining course at UT Austin. This project focused on analyzing data using the Darwin platform & tools. The group analyzed a large data set containing information on movies, and built models which would predict if a film would earn at least $100 million domestically.

Through the use of Jupyter Notebook, we were able to implement the entire project, from data collection, to the final modal & prediction. Our group began by collecting our data together and prepping it, removing any irrelevant columns. After this, we spent awhile simply exploring the data to get to know it a bit better. From here we were able to visually see binning based on domestic gross as well as a positive correlation between budget & domestic gross.

After exploring our data, we began to feature engineer our data, adding in new features based on pre-existing ones, such as combined income for actors, directors, and writers in a film. After this, we fed our data model to the Darwin SDK, which in turn created & trained our model for us. From here, we were able to get our feature importance listed as well as the final results of its prediction accuracy.
